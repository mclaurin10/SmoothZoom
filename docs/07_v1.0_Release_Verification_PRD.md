# v1.0 Release Verification & Certification

## macOS-Style Full-Screen Zoom for Windows

### Document 7 — Development Plan Series (Phase 7 PRD)

**Version:** 1.0
**Status:** Active (current phase)
**Last Updated:** 2026-06-18
**Prerequisites:** Document 1 — Project Scope (v1.2), Document 2 — Behavior Specification (v1.1), Document 4 — Phased Delivery Plan (v1.1, §9A)

> **One line:** Phases 0–6 delivered the complete v1.0 feature set; this phase verifies and certifies that feature-complete build for release. No new features.

---

## 1. Purpose & Context

The v1.0 feature set is **done** — all ten components and the full Phase 6 hardening set (color inversion, multi-monitor, crash recovery, conflict detection, hook watchdog, config schema versioning) are implemented and merged to `master`. Unit tests pass (pure-logic layers). What stands between the current build and a shippable **v1.0** is not implementation — it is *verification*:

- The **performance audit** (Doc 4 E6.12) has only been spot-checked for CPU on dev hardware. GPU, memory, and frame-latency have not been measured, and nothing has been measured on the Intel UHD 620 reference machine.
- The **139-acceptance-criteria sweep** (Doc 4 E6.13) is mostly unrecorded against the signed, installed build. Automated unit tests cover the pure-logic ACs; the manual ACs (focus/caret following, multi-monitor, error handling, settings lifecycle) need a recorded pass.

This PRD defines that verification campaign, its exit gate, and the artifacts that constitute "v1.0 certified." It is the Phase 7 entry in the Phased Delivery Plan (Doc 4 §9A); this document is the authority for the detail. The **live results tracker** is [`manual_test_checklist_v2.md`](manual_test_checklist_v2.md).

---

## 2. Scope of Verification

1. **Performance audit (E6.12)** — all five performance targets from Scope §6.2, measured on the reference machine and on dev hardware. (§5)
2. **Acceptance-criteria sweep (E6.13)** — all 139 non-deferred ACs from Document 2, verified PASS against the signed build, results recorded. (§6)
3. **Interactive / elevated test matrix** — the scenarios that require real hardware or a higher integrity level than automated tooling can drive: multi-monitor, mixed DPI, monitor unplug, UAC / secure desktop, lock/unlock, sleep/resume, native-Magnifier conflict, live crash recovery. (§7)
4. **Carried-over findings** — confirm-or-close the items earlier ad-hoc testing flagged. (§8)
5. **Release artifacts** — signed binary, completed checklist, verification report, version stamp. (§12)

---

## 3. Out of Scope

- **New features.** This phase changes behavior only to fix verification-blocking bugs.
- **Phase 8 (Input Interoperability).** Scroll-sensitivity/momentum settings UI, consolidating Raw Input/PTP into the Input layer, per-device robustness, and the pinch-feasibility spike are deferred until v1.0 ships (Doc 4 §9B, `input-interop-handoff.md`).
- **Desktop Duplication migration (R-01).** The capture backend stays on the Magnification API. The migration would unlock the deferred ACs (§9) but is a major post-1.0 effort.

## 4. Verification Approach & Environment

**Build under test.** A signed **Release** build installed to `C:\Program Files\SmoothZoom\` with the `uiAccess="true"` manifest. The full-screen Magnification API silently returns `FALSE` without all three of {signed, secure folder, UIAccess}, so verification must run against the installed binary — never a loose `build/` output. (R-12)

**Toolchain note.** `cmake`/`ctest` are not on `PATH` and the existing `build/` dir is stale. Build with the Visual Studio 2026 CMake directly into a fresh dir:

```powershell
$cmake = "C:\Program Files\Microsoft Visual Studio\18\Community\Common7\IDE\CommonExtensions\Microsoft\CMake\CMake\bin\cmake.exe"
& $cmake -S "C:\Dev\SmoothZoom" -B "C:\Dev\SmoothZoom\build2" -G "Visual Studio 18 2026" -A x64 `
    -D "FETCHCONTENT_SOURCE_DIR_CATCH2=C:/Dev/SmoothZoom/build/_deps/catch2-src"
& $cmake --build "C:\Dev\SmoothZoom\build2" --config Release
# Unit tests (pure logic): & "C:\Dev\SmoothZoom\build2\Debug\smoothzoom_tests.exe"
```

Then sign (`scripts\sign-binary.ps1`) and install (`scripts\install-secure.ps1`).

**Methodology.**
- *Automated logic ACs* — `ctest` / `smoothzoom_tests.exe` (Catch2). These cover ZoomController, ViewportTracker, WinKeyManager, SettingsManager, ScrollNormalizer, ModifierUtils, rect validation, SeqLock.
- *Driven manual ACs* — inject input via `SendInput` / `keybd_event` (reaches the low-level hooks; note that a generic automation tool's synthetic scroll does **not**) and observe the magnified output via screenshots (which **do** capture the DWM Magnification overlay).
- *Interactive/elevated ACs* — performed by a human on real hardware at the correct integrity level (a medium-IL shell cannot drive a `uiAccess` process or inject over a higher-IL foreground window; multi-monitor and secure-desktop scenarios need physical setup).
- *Performance* — build with the `SMOOTHZOOM_PERF_AUDIT` option for QPC frame-timing stats; use Performance Monitor (60 s samples), Task Manager working set, and GPU-Z / Intel Graphics Monitor.

**Recording.** Every result is logged in [`manual_test_checklist_v2.md`](manual_test_checklist_v2.md) with the date and the hardware it was observed on (dev vs. reference). Provisional results are labeled as such until confirmed on reference hardware.

## 5. Performance Audit (E6.12 → VER-2)

Target hardware: Intel UHD 620 graphics, 8 GB RAM, Windows 10 22H2 (Scope §6.2 / §6.4). Also recorded on dev hardware for comparison.

| Metric | Procedure | Target | Status (2026-06-18) |
|--------|-----------|--------|---------------------|
| Scroll-to-display latency | QPC stats from `SMOOTHZOOM_PERF_AUDIT`; cross-check with `DwmGetCompositionTimingInfo` | ≤16 ms at 60 Hz | Not yet measured |
| CPU idle (zoomed, pointer still) | Performance Monitor, 60 s | <2% | Dev hw: 0.00% (provisional) |
| CPU active (zoomed, pointer moving) | Performance Monitor, 60 s | <5% | Dev hw: 0.06% (provisional) |
| GPU overhead | GPU-Z / Intel Graphics Monitor at 5× | <10% on Intel UHD 620 | Not yet measured |
| Memory footprint | Task Manager working set during active zoom | <50 MB | Not yet measured |

If the idle-CPU target is missed on reference hardware, implement the **R-18 adaptive frame-rate** contingency (drop to ~60 Hz when no input is occurring, ramp to full VSync only during active zoom/pointer movement) and re-measure.

## 8. Carried-Over Findings to Confirm or Close

Earlier ad-hoc testing (archived `manual_test_checklist.md`) surfaced items to resolve in this phase. Each must end this phase as PASS, fixed, or a documented known-limitation:

- **AC-2.1.11** — occasional viewport drift/shift away from the pointer at zoom. (Viewport-drift fixes have since landed — re-verify; add diagnostic logging if it recurs.)
- **AC-2.1.16 / R-06** — Start Menu *sometimes* still opens on Win release after zooming. Intermittent suppression failure to reproduce and fix or document.
- **AC-2.1.19** — confirm a settings *modifier change* takes effect live (earlier report: "no apparent effect"). The keyboard hook now reads the live modifier; verify end-to-end.
- **E6.8** — after the native-Magnifier conflict dialog closes the other magnifier, scroll-to-zoom reportedly did not function. Verify the post-conflict init path.
- **AC-2.1.20/21** — reconcile against the Ctrl-exclusion policy (Doc 2 §15.2); these are reframed, not failures.

---

## 9. Deferred Items — Excluded From the Pass Bar (→ VER-5)

These are documented R-01 deferrals (Desktop Duplication migration), **not** test failures:

- **AC-2.3.08** — nearest-neighbor (sharp-pixel) rendering. The Magnification API always applies bilinear filtering.
- **AC-2.3.09 / AC-2.9.07** — image-smoothing toggle. No filtering parameter exists on `MagSetFullscreenTransform`; `imageSmoothingEnabled` is retained in the schema for forward-compatibility.
- **AC-MM.01, secondary clause** — "other monitors display at 1.0×." The API applies one global transform across the virtual desktop (same as the native Magnifier). The *primary* AC-MM.01 requirement (zoom active and tracking on the pointer's monitor) **is** in the v1.0 pass bar.

## 10. Exit Criteria — The Phase Gate

| # | Criterion |
|---|-----------|
| **VER-1** | All 139 non-deferred ACs verified PASS against the signed build; results recorded in `manual_test_checklist_v2.md`. |
| **VER-2** | Performance audit complete on the Intel UHD 620 reference machine **and** dev hardware, all five metrics meeting target (§5). Adaptive frame-rate (R-18) implemented if the idle target is missed. |
| **VER-3** | The full interactive/elevated matrix (§7) passes. |
| **VER-4** | Crash recovery, graceful exit, and hook-watchdog reinstall confirmed live on the signed build. |
| **VER-5** | Deferred items (§9) documented and excluded from the pass bar — confirmed not regressed, not counted as failures. |
| **VER-6** | Release artifacts produced (§12). |

v1.0 is **certified** when VER-1 through VER-6 are all satisfied and signed off.

---

## 11. Risks for This Phase

- **R-18 (performance on reference hardware)** — the headline risk; mitigation (redundant-frame skip) is in place and the adaptive frame-rate contingency is in reserve.
- **Reference-hardware availability** — VER-2 requires access to an Intel UHD 620 (or equivalent integrated-graphics) machine; secure VS Code time on one.
- **UIA cross-application variance (R-09)** — focus/caret following will not be perfect everywhere; the spec already defines "must work" vs. "best effort" tiers (Doc 4 §6.3.5) and graceful degradation, so document rather than block on best-effort apps.
- **Secure-desktop limits (R-15)** — accepted platform limitation shared by all magnifiers; verify graceful handling, not magnification on the secure desktop.
